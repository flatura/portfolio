# Роль и обязанности
## Моя роль

Я выступал как технический владелец архитектуры лабораторного PoC.

Моя зона ответственности:

- перевод продуктовой идеи в архитектурную модель;
- формулировка концепция контролируемого цикла работы с LLM;
- выбор временного стека для лаборатории PoC;
- проектирование синтетического датасета;
- проектирование диагностики, базирующейся на плейбуках;
- проектирование подхода использования инструментов;
- подготовка контрактов инструментов и мерик инструментов;
- проектирование доказателнього лога запуска;
- стабилизация demo-flow, дебаг;

### Выполненные работы

- Спроектировал общую архитектуру PoC: первичная оболчка в виде чата, сценарий работы агентов через backend-обвязку, роутинг плейбуков, слой инструментов, слой данных, слой документальной доказательности, трассировка запуска.
- Построил лабораторный agent runtime на FastAPI + LangGraph.
- Спроектировал принцип: LLM не обращается напрямую к PostgreSQL, Qdrant или MinIO.
- Реализовал/заложил контролируемый запуск инструментов через FastAPI tool-server.
- Подготовил синтетический датасет для fashion/retail/manufacturing-компании.
- Расширил датасет в сторону кросс-доменной диагностики: finance, delivery, ITSM, PMO, meetings, documents.
- Зафиксировал целевое направление: исполнительские отчеты, граф доказательств, проактивные уведомления, оркестрация через backend.

### Применение ИИ

Проект разрабатывался с использованием AI-assisted development.

LLM использовались для ускорения:

- генерации черновиков кода;
- подготовки synthetic data;
- подготовки промтов и логики процессов;
- анализа вариантов архитектуры;
- подготовки документации;
- генерации тестовых сценариев и проверок.

Ключевые решения оставались под ручным контролем:

- архитектурные границы;
- модель доступа к данным;
- выбор лабораторного и целевого runtime;
- требования к проверяемости;
- структура синтетического датасета;
- смысл плйбуков;
- ограничения PoC;
- ревью результата;
- позиционирование проекта.
