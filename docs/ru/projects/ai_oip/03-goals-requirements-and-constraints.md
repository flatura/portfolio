# Цели, требования и ограничения
## Цели и нецели
### Основные цели проекта

- Проверить, может ли LLM-агент работать с корпоративными цифровыми следами через контролируемые инструменты, а не через прямой доступ к данным.
- Построить воспроизводимый лабораторный стенд с синтетическими enterprise-данными.
- Реализовать первый вертикальный срез: вопрос пользователя -> выбор playbook -> вызов tools -> расчёты/RAG -> структурированный ответ -> evidence trail.
- Показать подход к управленческой диагностике, где выводы опираются на факты, документы, расчёты и явно указанные ограничения анализа.
- Подготовить архитектурную основу для будущего продукта: backend control plane, Tool Gateway, Tool Registry, playbooks, semantic layer, audit trail.

### Что не входит в текущий PoC

- Не является production-ready enterprise-платформой.
- Не является автономным "ИИ-директором".
- Не является заменой BI.
- Не является полноценным process mining продуктом.
- Не содержит реальных клиентских интеграций.
- Не является зрелой многотенантной имплементацией.
- Не содержит универсальный коннектор к корпоративным данным.
- Не является готовым локальным установщиком под ключ.
- Не содержит полноценный импортёр для данных из SharePoint, Jira, Confluence, Email и корпоративных DMS.

## Бизнес-требования

- **BR-001. Проверяемая управленческая аналитика** 
  Система должна помогать руководителю или владельцу бизнес-домена исследовать управленческий вопрос и получать вывод, опирающийся на расчёты, документы, ответы инструментов и явно указанные ограничения анализа.

- **BR-002. Снижение трудозатрат на подготовку аналитики**
  Система должна сокращать время первичного анализа управленческой гипотезы за счёт автоматизированного выбора сценария(плейбука), вызова инструментов, сбора доказательств и формирования ответа исполнительского уровня.

- **BR-003. Прозрачность вместо "магического ИИ"**
  Пользователь должен видеть не только итоговый текст, но и основание вывода: выбранный плейбук, вызванные инструменты, параметры, результаты, источники и ограничения.

- **BR-004. Безопасная работа с корпоративными данными**
  Система должна быть спроектирована так, чтобы LLM не получала прямой доступ к базам данных, документам, формированию произвольных SQL-запросов.

- **BR-005. Доменная управленческая диагностика**
  Система должна поддерживать диагностику не как свободный чат с данными, а как набор доменных плейбуков: финансовый анализ, операционная диагностика, PMO/ITSM/delivery-анализ, анализ целей и KPI.

- **BR-006. Демонстрация без клиентских данных**
  PoC должен демонстрировать ценность подхода на синтетическом enterprise-датасете, не требуя подключения реальных корпоративных данных клиента.

- **BR-007. Возможность расширения на новые бизнес-домены**
  Архитектура должна позволять добавлять новые домены анализа через плейбуки, инструменты, семантический слой и реестр инструментов, а не через ручное переписывание одного общего промта.

- **BR-008. Подготовка к enterprise-внедрению**
  Архитектурные решения PoC должны учитывать будущие требования enterprise-среды: аудит рассуждений, RBAC/ACL, контролируемые интеграции, on-prem/private развертывание, локальную LLM, воспроизводимость стенда и переносимость логики.

## Функциональные требования

- **FR-001. Чат-интерфейс для управленческого вопроса**
  Пользователь должен иметь возможность задать управленческий вопрос в свободной форме через чат-интерфейс.

- **FR-002. Выбор диагностического плейбука по входящему сообщению**
  Система должна определять домен вопроса и выбирать подходящий диагностический плейбук, иначе уточнить у пользователя, к какому домену относится его вопрос.

- **FR-003. Ограничение инструментов через плейбук**
  Выбранный плейбук должен ограничивать набор доступных инструментов, чтобы LLM не работала со всем каталогом инструментов сразу.  
  
- **FR-004. Доступ к данным только через tool-server**
  LLM должна получать данные через контролируемые инструменты, вызываемые через Сервер инструментов.  

- **FR-005. Структурированный ответ инструментов**
  Tool-server должен возвращать структурированный ответ (JSON) с результатом, метаданными, статусом выполнения и технической информацией, достаточной для лога и доказательства.  

- **FR-006. Расчёт финансовых метрик**
  Финансовые метрики должны рассчитываться по структурированным данным из БД, а не генерироваться LLM.  

- **FR-007. Использование RAG как документального доказательства**
  Система должна уметь использовать документы как дополнительный источник доказательств решения через MinIO/Qdrant/RAG-контур.  

- **FR-008. Структурированный ответ исполнительского уровня**
  Итоговый ответ должен включать факты, интерпретацию, гипотезы, ограничения анализа и рекомендуемые управленческие действия.  

- **FR-009. Прозрачность системы**
  Система должна раскрывать выбранный плейбук, запуск инструментов, входные параметры, ответы и промежуточные детали запуска.  

- **FR-010. Ответ на языке вопроса пользователя**
  Система должна формировать ответ на том же языке, на котором задан вопрос.  

- **FR-012. Уточняющий вопрос**
  Если вопрос неполный, слишком широкий или не содержит критичных параметров, система должна уточнить необходимую информацию, период, домен или цель анализа.

- **FR-013. Воспроизводимый импорт синтетических данных**
  Система должна поднимать demo-состояние из репозитория: PostgreSQL seed, MinIO объекты, Qdrant веса, манифесты и скрипты.  

- **FR-014. Поддержка нескольких доменов синтетических данных**
  Демо датасет должен содержать связанные домены: finance, sales, products, documents, delivery, ITSM, PMO, meetings и tasks как минимум на два года.

## Правила и ограничения

- **CON-001. Проверяемость выводов**
  Каждый существенный вывод должен иметь связь с ответом инструмента, документом, расчётом или явно указанным лимитом.

- **CON-002. Контролируемый доступ к данным**
  LLM не должна получать прямой доступ к PostgreSQL, Qdrant, MinIO, файловой системе или произвольному SQL.

- **CON-003. Трассируемость выполнения**
  Каждый запуск должен сохранять выбранный плейбук, запуск инстурментов, параметров, вответов, доказательств и финальный ответ.

- **CON-004. Только синтетические данные в PoC**
  В PoC используются синтетические данные. Реальные корпоративные данные клиентов не подключаются.

- **CON-005. Ограниченный масштаб PoC**
  PoC рассчитан на демонстрационные сценарии и архитектурную проверку, а не на промышленную нагрузку, массовых пользователей или любых SLA.

- **CON-006. Ограничение использования LLM**
  Использование локальной LLM является целевым для продукта, но для PoC допустимо использование внешней LLM. Система должна быть адаптированной к смене LLM.

## Ограничения и допущения

- Система работает на синтетическом датасете, а не на боевых данных клиента.
- Часть cross-domain сценариев демонстрационная и требует стабилизации.
- RAG-контур показывает подход к document evidence, но не является зрелым ingestion/lifecycle решением.
- Tool Registry и Tool Factory находятся в ранней стадии: часть реализована, часть описана архитектурно.
- UI является временным chat harness, а не целевым executive cockpit.
- Система не доказывает полноценную готовность к enterprise deployment; она доказывает архитектурную реализуемость controlled LLM analytics loop.
