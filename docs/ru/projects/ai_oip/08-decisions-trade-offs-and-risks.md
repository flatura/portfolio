# Решения, компромиссы и риски

## Ключевые решения

### 1. Backend как control plane

Backend должен управлять orchestration, tools, permissions, audit, semantic layer и reports. LLM не является системой доступа к данным.

### 2. LangGraph как lab runtime, не production core

LangGraph выбран для быстрого прототипирования stateful workflow, branching и tool loop. Целевая логика должна быть переносимой через FlowSpec в backend-native runtime.

### 3. Open WebUI как временный интерфейс

Open WebUI используется для демонстрации chat-flow. Целевой UI должен быть executive cockpit с report view, signal cards, evidence view и audit trail.

### 4. Playbooks вместо свободного набора tools

Система не раскрывает LLM все tools сразу. Playbook определяет доменную рамку и allowed tools.

### 5. Synthetic data вместо реальных корпоративных данных

PoC использует synthetic enterprise dataset, чтобы безопасно демонстрировать cross-domain диагностику без доступа к боевым данным.

### 6. RAG как document evidence, а не замена SQL

Структурированные метрики считаются в PostgreSQL/tools. RAG используется для документов, решений, отчётов, протоколов и контекста.

## Architecture Decision Records

### 1. Интерфейс

#### Контекст
Для PoC нужно быстро показать основной пользовательский сценарий: руководитель задаёт управленческий вопрос в свободной форме, система выбирает диагностический плейбук, вызывает инструменты и возвращает проверяемый ответ с доказательным логом.

#### Принятое решение
Open WebUI как временный интерфейс для демонстрации PoC.

#### Отклонённая альтернатива
Разработка собственного UI с нуля на старте потребовала бы отдельного фронтенд-цикла: проектирования UX, авторизация, модель доступа, истории переписки. Это увеличило бы сроки и отвлекло бы от проверки главной архитектурной гипотезы.

#### Обоснование

- Open WebUI позволяет быстро получить рабочий chat-интерфейс без затрат на frontend-разработку.
- Для PoC важнее проверить контролируемую аналитику средствами LLM, чем финальный пользовательский интерфейс.
- Open WebUI достаточно для демонстрации базового сценария: предоставить интерфейс чата, передать вопрос на обработку, получить ответ и отобразить пользователю.
- Временный harness снижает стоимость экспериментов и ускоряет обратную связь по продуктовой гипотезе.

#### Компромиссы

- Детали, трассировку и доказательства необходимо генерировать на стороне backend и отображать в новой вкладке браузера по прямой ссылке в ответе. 
- Open WebUI не показывает целевой исполнительский UX.
- Для enterprise-продукта потребуется собственный интерфейс: дашборд индикаторов, карточки рисков, блоки доказательств, историю запусков и управляемая навигация по источникам.
- Есть риск, что целевая аудитория PoC будет воспринимать Open WebUI как "ещё один чат с LLM".

### 2. Стек для обвязки и бизнес-логики (harness)
#### Контекст

PoC требует stateful процесса на каждый запрос: планирование шага анализа, вызов инструментов, оценка достаточности доказательств, возможный перезапуск инстурментов с другими параметрами, финализация ответа. Нужен быстрый способ отлаживать агентный цикл, раз, состояние запуска и промт-итерации.

#### Принятое решение
LangGraph + FastAPI

#### Отклонённая альтернатива

- Java Spring Boot backend
- n8n - не даёт достаточной гибкости в сложных ветвлениях, динамическом формировании агентного контекста, сохранении контекста для асинхронной работы. Больше приспособлен к статичным автоматизациям без условных ветвлений.
- Dify - оверинжениринг для PoC, тяжеловесная платформа (8+ контейнеров в минимальной поставке), недостаточно гибкости на этапе разработки, медленный и сложный дебаггинг

#### Обоснование

- LangGraph быстро даёт рабочую модель агентного процесса работы, используя ноды: планировщик -> запуск инструмента -> оценка достаточности -> продолжение планирования/финализация -> формулирование ответа и отчета. 
- Удобно отлаживать состояние run, переходы, prompt-логику и поведение LLM.
- FastAPI-обёртка позволяет быстро предоставить конечную точку для Open WebUI и smoke-тестов.
- LangGraph подходит для лабораторной проверки ответвлений и agent-loop гипотез.
- Workflow-логика может быть дополнительно описана через FlowSpec, чтобы позже перенести её на целевой стек.

#### Компромиссы

- LangGraph не должен становиться финальным production-ядром.
- Логику workflow придётся переносить в продуктовый backend.
- Есть риск зависимости от Python/LangGraph-специфики, если своевременно не фиксировать framework-neutral FlowSpec.
- Production control plane должен управлять permissions, audit, tool policy, reports и orchestration централизованно, а не делегировать это лабораторному runtime.

### 3. Синтетический датасет вместо реальных клиентских интеграций
#### Контекст

Для демонстрации Executive Decision Intelligence нужны связные данные из нескольких доменов: finance, sales, delivery, ITSM, PMO, meetings, documents. Реальные корпоративные данные получить сложно: они чувствительные, грязные, неполные, требуют согласований, безопасности и интеграционной работы.

На этапе PoC нужно доказать архитектурную реализуемость подхода без зависимости от клиента и без риска утечки данных.

#### Принятое решение

Использовать синтетический корпоративнй датасет с заранее заложенными crossкросс-доменными сценариями и контролируемыми аномалиями.

#### Отклонённая альтернатива

Подключать реальные клиентские системы: ERP, CRM, Jira/YouTrack, ITSM, SharePoint, Confluence, Email, Calendar, DMS.

#### Обоснование

- Синтетические данные безопасны для демонстрации и публикации в портфолио.
- Датасет можно воспроизводимо поднимать из репозитория
- В синтетике можно заложить контролируемые аномалии: известные скрытые причины, ожидаемые выводы.
- Такой датасет позволяет тестировать не только правильный путь, но и способность системы находить заранее заложенную управленческую причинность.
- Демо не зависит от доступности клиента, NDA, разрешений и качества реальных интеграций.

#### Компромиссы

- PoC не проверяет поведение системы на грязных, неполных и противоречивых реальных данных.
- Не проверены реальные интеграционные проблемы: ACL, версионирование документов, свежесть источников, нестабильные API, неполные выгрузки, дубли и расхождения.
- Синтетика может выглядеть слишком аккуратной по сравнению с enterprise-реальностью.
- Для пилота потребуется отдельный этап изучения корпоративных систем, взаимосвязи этих систем в разрезе бизнес-процессов, проверку качества данных и настройка семантического слоя под клиента.

### 4. Доступ к источникам данных
#### Контекст

Ключевой архитектурный принцип PoC: LLM не получает прямой доступ к данным. Все обращения к PostgreSQL, Qdrant и MinIO должны идти через оркестратор и сервер инструментов.

#### Принятое решение
Легковесный FastAPI tool-server

#### Отклонённая альтернатива

- Оставить прямой доступ к источникам данных, реализовав инструменты непосредственно в обвязке - противоречит ключевой архитекурной концепции, ИБ заказчика не пропустит
- Tool Gateway продуктового уровня на Java - преждевременно и трудоемко

#### Обоснование

- FastAPI позволяет быстро реализовать контролируемые инструменты
- Абстракция от обвязки уже на текущем этапе позволит развивать сервер отдельно от обвязки, а в последствии эволюционно развить в Tool Gateway: добавить реестр инструментов, ролевую модель, аудит-лог, лимитер.
- Каждый tool имеет явный contract: input, validation, output, metadata.
- Такой слой достаточен для проверки главной гипотезы: LLM выбирает действие, backend исполняет, результат возвращается как evidence.

с моделью доступа, политикой аудита, регистром инструментов, процессом подтверждения, менеджментом инструментов и полноценной моделью безопасности.

#### Компромиссы

- Реализация не является зрелой моделью.
- Нет полноценного RBAC/ACL на корпоративном уровне.
- Не предполагается проверка на потенциально опасные действия.
- Нет рейт-лимита, управления квотами на использование источников данных.

### 5. Работа с LLM API

#### Контекст

PoC требует качественного размышления, структурированного вывода и быстрой итерации. Локальная модель потребовала бы подбора железа, настройки inference-сервера, оптимизации latency, тестирования качества и решения проблем model serving.

На этапе PoC важнее отладить и проверить архитектуру контролируемого цикла LLM размышлений, чем доказывать локальное развёртывание модели.

#### Принятое решение
Внешний OpenAI-compatible LLM API

#### Отклонённая альтернатива

- Развернуть локальную модель через vLLM, Ollama, llama.cpp или другой OpenAI-compatible inference endpoint - нет доступного железа + необходима возможность переноса и проверки на другом демо-железе
- Развернуть лок.модель на арендованном сервере - отложено до более поздних, пост PoC вех.

#### Обоснование

- Внешний API быстрее подключить и проще стабилизировать на этапе PoC.
- Качество рассуждений и следования инструкциям обычно выше, чем у быстро поднятой локальной модели.
- OpenAI-совместимый интерфейс сохраняет возможность позже заменить провайдера на локальную модель.
- PoC не работает с реальными клиентскими данными, поэтому риск использования внешнего API ниже.
- Решение более простое, позволяет сосредоточиться на других аспектах требований

#### Компромиссы

- Возникает зависимость от внешнего API, стоимости, задержке и доступности провайдера.
- Для enterprise-клиентов использование внешней LLM может быть неприемлемо из-за политики данных.
- В production потребуется доступный механизм переключения между моделями.
- Для реальных корпоративных данных понадобится строгая политика: какие данные можно отправлять в модель, какие должны оставаться внутри периметра, какие сценарии требуют локальной модели.

## Основные риски

1. Scope explosion - продукт легко расползается в finance, process mining, goal-setting, personal assistant, meeting transcription и daily reports. Нужен жёсткий вертикальный срез по бизнес-процессу.
2. Недостаток данных - реальные компании могут не иметь нужных цифровых следов или доступ к ним будет политически закрыт.
3. Неполная проверяемость - без связи с результатами вызова инструментов ответ может выглядеть убедительно, но быть слабо проверяемым (это свойство LLM GPT).
5. Корпоративная безопасность - production-версия потребует отдельной работы по RBAC, ACL, аудиту, секретам, развертке и сопровождении.
