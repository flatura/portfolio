# MyLovelyGarden (MLG): B2B SaaS ERP для ботанических организаций

**Статус:** MVP (Подготовка к пилоту)
**Роль:** Solo Founder / Solution Architect / AI-Augmented Engineer
**Платформа:** Web (SPA)

## О продукте
**MyLovelyGarden** - это комплексная ERP-система, спроектированная для ботанических садов, питомников, селекционеров и коллекционеров.

**Бизнес-проблема:** Исторически учет биологических фондов ведется в разрозненных Excel-таблицах или устаревшем десктопном и дорогостоящем ПО. Это делает невозможным сквозной поиск, обмен коллекциями между организациями и точную гео-привязку экземпляров.
**Решение:** Единая SaaS-платформа с централизованным справочником таксонов (более 1.5 млн видов), строгой ролевой моделью и профессиональным ГИС-модулем для картографирования посадок.

---

## Архитектурное видение и AI-Augmented подход

Продукт разрабатывается с применением методологии **AI-Assisted Full-Stack Generation**. В качестве Solution-архитектора я проектирую спецификации, API-контракты и модели данных, а генерация рутинного кода (как на бэкенде, так и на фронтенде) делегируется LLM-агентам (Cursor, Claude).

Этот подход (Spec-Driven Development) позволил сократить Time-to-Market (TTM) MVP-версии в несколько раз, осуществив успешную миграцию с legacy-монолита на современную Decoupled-архитектуру всего за 2 недели.

## Ключевые архитектурные решения (ADR Highlights)

Ниже представлены некоторые концептуальные решения, принятые в ходе проектирования системы (полный реестр ADR ведется в закрытом репозитории продукта).

### 1. Архитектура приложения (Decoupled SPA)
* **Решение:** Отказ от Server-Side Rendering (Thymeleaf) в пользу разделения на stateless Backend REST API (Java 21, Spring Boot 3.3) и Frontend SPA (Angular 21).
* **Обоснование:** Использование Signals и Zoneless-архитектуры в Angular в связке со строгой типизацией TypeScript обеспечивает высокую скорость UI-рендеринга и идеально ложится на паттерны AI-кодогенерации. REST API закладывает фундамент для будущих мобильных клиентов.
* [Читать полный ADR-001 (Смена технологического стека)](mlg-adr-001.md)

### 2. Изоляция данных (Soft Multi-tenancy)
* **Решение:** Реализация паттерна Soft Multi-tenancy с использованием Discriminator Column (`root_unit_id`) и фильтрацией на уровне Hibernate и Security Context.
* **Обоснование:** Выбор сделан в пользу единой схемы БД (в противовес Database-per-tenant) для снижения стоимости владения (TCO) на этапе старта и обеспечения возможности сквозной глобальной аналитики по видам растений без сложных ETL-процессов. Угрозы утечки данных нивелируются на уровне ORM.

### 3. Гибридная ролевая модель (Context-Aware RBAC)
* **Решение:** Внедрение двухуровневой системы авторизации (System Roles + Organization Roles).
* **Обоснование:** Классическая "плоская" модель не подходит для B2B SaaS. Пользователь может быть системным администратором платформы, Главным ботаником в Саду "А", стажером в Саду "Б" и не иметь доступа к Саду "В". Security Context валидируется динамически в зависимости от выбранного `root_unit_id`.

### 4. Архитектура геоданных (Enterprise GIS)
* **Решение:** Использование `PostgreSQL + PostGIS` на бэкенде (типы `GEOMETRY` с проекцией 4326) и полнофункционального фреймворка `OpenLayers` на фронтенде.
* **Обоснование:** Требуется высокая точность (R-Tree индексы на уровне БД) для тяжелых пространственных запросов ("найти все виды растений в заданном полигоне"). Выбор OpenLayers обусловлен нативной поддержкой сложного редактирования полигонов (Draw, Snap, Modify) без зависимости от сторонних плагинов.

### 5. Жизненный цикл сущностей (Event Sourcing Lite)
* **Решение:** Внедрение гибридной модели для учета растений. Лог всех действий пишется в отдельную иммутабельную таблицу `plant_events`, а актуальное состояние для быстрого чтения проецируется в плоскую таблицу `plant_instances`.
* **Обоснование:** Удовлетворяет двум полярным бизнес-требованиям: научным сотрудникам нужен полный аудит истории растения, а садовникам - мгновенная выборка текущего состояния для работы в полях.

## Инфраструктура и DevOps (Transitional Architecture)

Стратегия развертывания построена по принципу "Минимальный TCO сейчас, готовность к Cloud-Native завтра".

* **Развертывание:** Production-среда развернута на VPS с использованием Docker Compose и Nginx (Reverse Proxy). Сознательный отказ от Managed Kubernetes (K8s) на этапе MVP для контроля операционных расходов.
* **Хранение медиа:** Внедрено S3-совместимое объектное хранилище `MinIO`. Вынос BLOB-данных из основной СУБД разгружает базу, а миграция в облако (AWS S3) в будущем потребует изменения лишь одной переменной окружения.
* **CI/CD:** Настроены пайплайны GitHub Actions для автоматической сборки Docker-образов (раздельно для API и SPA). Деплой и запуск пока в ручном режиме для проверки успешности запуска.
* **Roadmap миграции:** При достижении лимитов вертикального масштабирования предусмотрен бесшовный перенос нагрузок в AWS (ECS Fargate + Amazon Aurora PostgreSQL + S3).
