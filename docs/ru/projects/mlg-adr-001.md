# ADR-001: Миграция с Server-Side Rendering монолита на Decoupled SPA архитектуру

**Статус:** Принято / Реализовано
**Проект:** MyLovelyGarden (SaaS ERP)
**Дата:** Октябрь 2025

## 1. Контекст и Проблема
Исторически MVP проекта был реализован как монолитное приложение на Spring Boot 2.7 с использованием Server-Side Rendering (Thymeleaf) для генерации UI. По мере усложнения бизнес-логики (интерактивные ГИС-карты, формы редактирования таксонов, фильтры ) связность UI и бэкенда стала критическим "бутылочным горлышком". Разработка сложных интерфейсов силами бэкенд-разработчика замедлилась, а кодовая база стала трудно поддерживаемой.

Возникла необходимость в переходе на стек, обеспечивающий высокую скорость вывода фич на рынок (Time-to-Market) для SaaS-пилота, с возможностью эффективного применения AI-инструментов кодогенерации.

## 2. Альтернативы
1. **Рефакторинг монолита:** Обновление до Spring Boot 3.3 и сохранение Server-Side Rendering (возможно, с интеграцией HTMX для реактивности).
2. **Decoupled Architecture (SPA + REST API):** Полное разделение на клиентскую часть (Angular, React/Vue) и stateless бэкенд.

## 3. Принятое решение
Осуществить переход на **Decoupled-архитектуру: Java 21 + Spring Boot 3.3 (Backend REST API) и Angular 21 (Frontend SPA)**. Разработку вести в парадигме AI-Assisted Development.

Для обеспечения безопасности API внедрить паттерн Chain of Responsibility (Auth Check через JwtFilter, Rate Limiting).

## 4. Обоснование (Pros)
* **Синергия с AI-инструментами:** Строгая типизация TypeScript (приближенная к Java) и новая парадигма Angular (Signals, Zoneless architecture) идеально подходят для генерации кода LLM. Это позволяет бэкенд-специалисту прототипировать сложный UI со скоростью Senior Frontend разработчика.
* **Производительность Backend:** Переход на Java 21 открывает доступ к Virtual Threads (Project Loom), что позволит платформе держать высокую конкурентную I/O нагрузку без усложнения кода реактивным стеком.
* **API-First подход:** Бэкенд становится чистым, переиспользуемым REST API. Это закладывает фундамент для будущей разработки нативных мобильных приложений без изменения серверной логики.

## 5. Компромиссы и Риски (Trade-offs & Cons)
* **Усложнение инфраструктуры доставки:** Вместо деплоя одного монолитного артефакта, CI/CD пайплайн усложняется: требуется раздельная сборка статики и JAR-файла.
* **Управление состоянием и безопасностью:** Вынужденный отказ от сессий (Session-based auth) в пользу внедрения JWT-авторизации (JwtFilter) и настройки CORS.
* **SEO Ограничения:** Потеря нативной индексации из-за отказа от SSR. В случае необходимости создания публичного каталога потребуется внедрение Angular SSR/Prerender.
* **AI-зависимость:** Высокая скорость AI-разработки требует жесткого архитектурного надзора и ревью кода, чтобы предотвратить деградацию качества кодовой базы.
